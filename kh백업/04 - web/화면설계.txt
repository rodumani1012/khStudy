UI의 개념
- 사용자와 컴퓨터가 정보를 주고받기 위해 사용자와 프로그램이 상호 작용하는 프로그램의 일부분으로 정보기기나 소프트웨어의 화면 등 사용자와 접하는 영역을 말한다.

요구사항 명세서 작성 원칙
- 시스템이 수행할 모든 기능과 시스템에 영향을 미치는 제약 조건을 명확하게 기술한다.
- 명세 내용은 고객과 개발자 사이에서 모두가 이해하기 쉽게 간결하게 작성한다.
- 기술된 모든 요구사항은 검증이 가능하도록 원하는 시스템의 품질, 상대적 중요도, 품질의 측정 및 검증 방법 및 기준 등을 명시한다.

UI 설계문서 작성 목적
- 커뮤니케이션 

UI 설계시 표준화 항목 대상 - 동선 규칙 - 레이블 규칙 - 화면 분할 규칙

UI 설계문서와 함께 작성되어야 하는 문서 - IA(information Architecture) - DFD(Data Flow Diagram) - FlowChart

UI 설계 문서에 포함되어야 하는 내용 - 제목 - 버전 - 프로젝트명

화면 설계 용어/정의 (다르게 연결된 정의는??)
-프로토타입. 시스템의 초기모델 -와이어프레임. 구조 컨텐츠 및 기능을 설명하는데 사용하는것 - 목업. 실제모형 -스토리보드. 기존의 아이디어를 세밀하게 묘사한 것

UI 설계문서에 반드시 담아야 할 내용(서술형)
1) 제목, 버전, 프로젝트명, IA번호, 페이지번호 등의 메타데이터
2) Wireframe : 화면(캔버스) 설계 
3) Storyboard : 화면 설명
4) Workflow : 이동 링크(네비게이션)
5) IA : 메뉴 구성도 (Information Architecture)

UI 설계시 고려사항
1) UI 화면 설계 작업을 실제 화면과 동일한 구도로 작업해야 한다.
화면설계시 실제 화면과 1:1 구도로 작업해야 화면 전체적으로 올릴 컨텐츠의 구조와 감이 잡히며, 디자이너와 커뮤니케이션이 원활해 진다.
2) 컨텐츠 구성 및 배치는 반드시 이유와 목적이 있어야 한다.
3) 사업의 전략, 컨셉, 서비스 컨셉, 메인-서브 페이지의 컨셉을 먼저 잡아놓고 UI 설계를 해야 한다.
4) 모든 기획은 사용자의 입장에서 행해져야 한다.
대부분의 기획자들이 사용자의 입장을 고려해 화면 설계를 한다고 하지만 진정으로 사용자들의 숨겨져 있는 니즈를 완벽하게 캐치하여 기획하는 기획자는 드물다. 구축하려는 사이트가 있다면 자신이 고객이 되어 직접 해 봐야 한다. 여러 상황을 직접 경험하고 고객과 감정이입이 되지 않는 이상 고객의 입장을 배려한 사이트는 나오기 힘들다.
5) UI 화면설계와 문서는 직관적이며 쉽게 구성되어야 한다.
모든 문서는 커뮤니케이션을 목적으로 하므로, 자신만 말아보는 말이나 패턴 등을 사용하면 여러 이해 관계자들돠의 원활한 커뮤니케이션을 방해하므로 개선해야 한다.
